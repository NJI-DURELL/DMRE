# DMRE — Hosting & Chrome Web Store Deployment

This guide walks through deploying the FastAPI backend to Render, pointing the
React dashboard and Chrome extension at it, and submitting the extension to
the Chrome Web Store. Each end user gets their own private memory store
(per-user isolation enforced server-side).

---

## 1. Deploy the backend to Render

The repo contains a top-level `render.yaml` (Blueprint) that creates one
web service + one managed Postgres + a persistent disk for ChromaDB and the
HuggingFace model cache.

1. Push this repo to GitHub (or GitLab / Bitbucket — anything Render
   supports).
2. Render → **New** → **Blueprint** → connect the repo. Render reads
   `render.yaml` and provisions the database, the disk, and the web service.
3. The first deploy:
   - builds `backend/Dockerfile`,
   - runs `alembic upgrade head` inside `backend/start.sh`,
   - boots Uvicorn on `$PORT`,
   - downloads Sentence-BERT (~80 MB) into `/data/hf` (cached after the
     first deploy thanks to the persistent disk).
4. Wait for the service to go green and copy the URL — e.g.
   `https://dmre-backend.onrender.com`.

### Verify the deploy

```bash
curl https://dmre-backend.onrender.com/health
# {"status":"ok","version":"1.0.0"}

# Sign up
curl -X POST https://dmre-backend.onrender.com/api/auth/signup \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com","password":"a-strong-password"}'
# {"access_token": "...", "user": {...}}
```

If `/health` returns 200 and signup returns a token, the backend is live and
multi-tenant.

### Production env vars worth tightening

Set these on the Render service (Settings → Environment):

| Var              | Recommendation                                                      |
|------------------|---------------------------------------------------------------------|
| `JWT_SECRET`     | Auto-generated by Render — leave alone unless you need to rotate.   |
| `CORS_ORIGINS`   | `https://YOUR-DASHBOARD.example.com,chrome-extension://<your-id>`   |
| `JWT_EXPIRE_MINUTES` | Default 30 days; lower if you want shorter sessions.            |

Leave `GANACHE_RPC_URL` / `CONTRACT_ADDRESS` blank — the hosted backend will
silently skip blockchain anchoring. You can still demo blockchain locally
during your defense.

## 2. Point the dashboard at the hosted backend

Build the dashboard with `VITE_BACKEND_URL` baked in:

```bash
cd dashboard
echo "VITE_BACKEND_URL=https://dmre-backend.onrender.com" > .env.production
npm install
npm run build      # outputs to dashboard/dist
```

Host `dashboard/dist` anywhere static (Render Static Site, Netlify, Vercel,
Cloudflare Pages, GitHub Pages). Add the resulting origin to
`CORS_ORIGINS` on the backend.

---

## 3. Configure the Chrome extension

Edit `extension/config.js`:

```js
export const BACKEND_URL   = "https://dmre-backend.onrender.com";
export const DASHBOARD_URL = "https://dmre.example.com";
```

Test the extension first by loading it unpacked:

1. `chrome://extensions` → enable **Developer mode**.
2. **Load unpacked** → pick the `extension/` directory.
3. Open the popup → sign up → browse a real page → confirm the popup shows
   `Last captured page: ok`.
4. Open the dashboard → sign in with the same email → search.

Once happy, copy the extension ID Chrome assigned (`chrome://extensions`)
and add `chrome-extension://<id>` to `CORS_ORIGINS` on the backend.

## 4. Submit to the Chrome Web Store

1. Pay the one-time $5 developer registration at
   <https://chrome.google.com/webstore/devconsole>.
2. Zip the `extension/` directory:
   ```bash
   cd extension
   zip -r ../dmre-extension-v1.0.0.zip . -x "*.DS_Store" "*.gitignore"
   ```
3. **New item** → upload the zip.
4. Fill in the listing:
   - **Description** — one paragraph for the store front and a longer
     detailed description.
   - **Screenshots** — 1280×800 PNGs of the popup signed-in, the dashboard
     search results, and (optionally) the verify-on-blockchain flow.
   - **Privacy policy URL** — required because the extension reads page
     content. Host a short policy that explains:
     - what is captured (URL, title, visible text, dwell, clicks, scroll),
     - where it is stored (your Render backend, per-user, encrypted at rest
       by Postgres),
     - that the user can sign out at any time and that they own their data.
   - **Permissions justification** —
     - `tabs`, `activeTab` — needed to know which page is active and to
       open the dashboard in a new tab.
     - `storage` — stores the JWT and the capture toggle locally.
     - `scripting` — bundled by MV3 even when not used directly.
     - `host_permissions: <all_urls>` — required because the user is
       indexing **their own** browsing history, which spans every site.
5. Submit for review. First review typically takes 1-3 days.

---

## 5. After publication

1. Replace the dev `chrome-extension://*` glob in `CORS_ORIGINS` with the
   real published ID.
2. Increase Render's plan if you expect concurrent users — the embedder
   needs ~1 GB RAM under load.
3. Backups: Render's managed Postgres has automatic daily backups on paid
   plans. The Chroma persistent disk does not — schedule a `tar` snapshot
   into S3 if you care.

## Troubleshooting

| Symptom                                              | Cause / Fix |
|------------------------------------------------------|-------------|
| Popup says "Backend offline"                         | `BACKEND_URL` in `extension/config.js` doesn't match the Render URL, or Render service is sleeping (free tier sleeps after 15 min idle). |
| Popup says "session expired"                         | `JWT_EXPIRE_MINUTES` elapsed. Sign in again. |
| Signup returns 422                                   | Password < 8 chars or username has invalid characters. |
| Search returns "not_found" for a page you visited    | Backend captured it under a *different* user. Ensure you're signed into the same account on the extension and the dashboard. |
| Render deploy fails on `alembic upgrade head`        | Database is empty but Alembic version table exists — open the Render shell and `psql $DATABASE_URL -c 'DROP TABLE alembic_version'`, then redeploy. |
